## 🔒 БЕЗОПАСНОСТЬ (ОБЯЗАТЕЛЬНО ДЛЯ ВСЕХ ПРОЕКТОВ)

### Критические правила безопасности

#### 1. Защита от XSS (Cross-Site Scripting)
- **НИКОГДА не используйте `innerHTML` или `dangerouslySetInnerHTML` без санитизации**
- Все пользовательские данные должны быть экранированы перед отображением
- Используйте функции из `src/lib/security.ts`: `escapeHtml()`, `sanitizeHtml()`, `sanitizeText()`
- При создании DOM-элементов используйте `createElement` и `textContent`, а не `innerHTML`

**Пример правильного подхода:**
```typescript
// ❌ НЕПРАВИЛЬНО
element.innerHTML = userInput; // ОПАСНО!

// ✅ ПРАВИЛЬНО
element.textContent = userInput; // Безопасно
// или
element.innerHTML = sanitizeHtml(userInput); // Если нужен HTML
```

#### 2. Валидация URL
- **ВСЕ URL должны проверяться перед использованием**
- Используйте `isValidUrl()` и `sanitizeUrl()` из `src/lib/security.ts`
- Разрешайте только безопасные протоколы: `http:`, `https:`, `data:` (только для изображений)
- **БЛОКИРУЙТЕ:** `javascript:`, `vbscript:`, `data:text/html` и другие опасные протоколы

**Пример:**
```typescript
const safeUrl = sanitizeUrl(userProvidedUrl);
if (!safeUrl) {
  // Показать ошибку, не использовать URL
  return;
}
img.src = safeUrl; // Безопасно
```

#### 3. Защита от скрытых вызовов к БД и внешним API
- **КРИТИЧНО:** При обработке текстовых данных (чтение/запись, парсинг JSON, обработка формул) **НИКОГДА не выполняйте скрытые вызовы к базам данных или внешним API**
- Все операции с данными должны быть явными и прозрачными
- Не используйте `eval()`, `Function()`, `import()`, `require()` в пользовательских данных
- Все формулы должны валидироваться через `isValidFormula()` перед выполнением

**Запрещённые паттерны:**
```typescript
// ❌ НЕПРАВИЛЬНО - скрытый вызов к БД в текстовой обработке
function processText(text: string) {
  // ОПАСНО: скрытый вызов к БД
  const result = db.query(text); // НИКОГДА так не делать!
  return result;
}

// ❌ НЕПРАВИЛЬНО - eval в пользовательских данных
const formula = userInput; // из пользовательского ввода
const result = eval(formula); // ОПАСНО!

// ✅ ПРАВИЛЬНО - явная валидация и безопасное выполнение
const validation = isValidFormula(userInput);
if (!validation.valid) {
  throw new Error(validation.error);
}
const result = evaluate(formula, safeScope); // math.js с безопасным scope
```

#### 4. Валидация всех пользовательских данных
- **ВСЕ данные от пользователя должны валидироваться перед использованием**
- Используйте `validateBlocks()` для схемы блоков
- Используйте `validateImportedBlocks()` для импортируемых JSON
- Проверяйте типы, обязательные поля, зависимости

**Пример:**
```typescript
// При импорте JSON
const validation = validateImportedBlocks(jsonString);
if (!validation.valid) {
  alert(`Ошибка: ${validation.error}`);
  return;
}
const blocks = validation.blocks; // Только после валидации
```

#### 5. Защита формул от вредоносного кода
- Все формулы проверяются на опасные паттерны:
  - `eval()`, `import()`, `require()`, `Function()`
  - Доступ к `constructor`, `__proto__`, `prototype`
- Используйте только `math.js` `evaluate()` с безопасным scope
- Никогда не передавайте пользовательские данные напрямую в `evaluate()` без валидации

#### 6. Защита при сохранении/загрузке
- Все данные валидируются перед сохранением в localStorage
- Все данные валидируются при загрузке из localStorage
- При ошибке валидации — не загружать данные, показать ошибку

#### 7. Санитизация при рендеринге
- Все текстовые блоки санитизируются через `sanitizeText()`
- Все URL проверяются через `sanitizeUrl()`
- Все HTML-контент экранируется через `escapeHtml()`

### Best Practices безопасности

1. **Принцип наименьших привилегий**
   - Давайте минимально необходимые права
   - Ограничивайте доступ к опасным функциям

2. **Явность лучше неявности**
   - Все операции с данными должны быть явными
   - Не скрывайте вызовы к БД/API в текстовых операциях

3. **Валидация на входе**
   - Валидируйте данные сразу при получении от пользователя
   - Не откладывайте валидацию

4. **Санитизация на выходе**
   - Санитизируйте данные перед отображением
   - Экранируйте HTML-специальные символы

5. **Логирование безопасности**
   - Логируйте все попытки импорта невалидных данных
   - Логируйте ошибки валидации

6. **Защита от инъекций**
   - Не используйте конкатенацию строк для SQL/NoSQL запросов (если будет БД)
   - Используйте параметризованные запросы
   - Валидируйте все входные данные

7. **Content Security Policy (CSP)**
   - Добавляйте CSP заголовки в HTML
   - Ограничивайте источники скриптов и стилей
