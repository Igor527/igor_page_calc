# Метеостанция: Google Таблица → Firebase — пошаговая настройка

Данные из Google Таблицы отправляются в **Firebase Realtime Database** скриптом **Apps Script**. Сайт читает лог по REST — без CORS и без публичной ссылки на таблицу.

---

## Часть A. Firebase

### A1. Проект и Realtime Database

1. Откройте [Firebase Console](https://console.firebase.google.com/).
2. **Добавить проект** (или выберите существующий).
3. В меню слева: **Build → Realtime Database**.
4. **Create Database**.
5. Режим на первое время можно выбрать **в тестовом режиме** (любой читает/пишет 30 дней) — ниже замените на правила с секретным путём.
6. Регион (например `europe-west1` или `us-central1`) запомнить не обязательно — важно скопировать **URL базы**.

**URL базы** выглядит так:

- Старый формат: `https://ИМЯ_ПРОЕКТА-default-rtdb.firebaseio.com`
- Новый (по региону): `https://ИМЯ_ПРОЕКТА-default-rtdb.REGION.firebasedatabase.app`

Скопируйте URL **без** завершающего слэша.

### A2. Секретный ключ (один на таблицу и на сайт)

Сгенерируйте длинную случайную строку (20+ символов). В терминале:

```bash
openssl rand -hex 32
```

Сохраните её — это `WEATHER_DATA_KEY` / `VITE_WEATHER_DATA_KEY`. **Не публикуйте** в открытом репозитории.

### A3. Правила Realtime Database

**Realtime Database → Rules** вставьте (доступ только к ветке `weather` под любым ключом):

```json
{
  "rules": {
    "weather": {
      "$dataKey": {
        ".read": true,
        ".write": true
      }
    }
  }
}
```

- Запись и чтение возможны по пути `weather/ВАША_СТРОКА_КЛЮЧА/rows/...`.
- Сайт и Apps Script знают ключ — посторонним без ключа путь не угадать (но для продакшена можно ужесточить правила через Firebase Auth — это отдельная настройка).

Нажмите **Publish**.

## Часть B. Сайт (.env)

В корне проекта в `.env` (или в секретах CI/Cloudflare):

```env
# URL Realtime Database (как в Firebase Console)
VITE_FIREBASE_DATABASE_URL=https://ВАШ_ПРОЕКТ-default-rtdb.firebaseio.com

# Тот же ключ, что WEATHER_DATA_KEY в Apps Script
VITE_WEATHER_DATA_KEY=ваш_секрет_из_openssl_rand

# Опционально — подпись на странице метеостанции
# VITE_WEATHER_STATION_ID=Станция_12
```

После изменения `.env` — **пересоберите** сайт (`npm run build`) и задеплойте.

Проверка: откройте **Метеостанцию** после входа — если конфиг есть, страница подтянет данные из Firebase (кнопка «Обновить» / авто-загрузка).

## Часть C. Google Таблица и Apps Script

### C1. Таблица с данными

- Лист с колонками по смыслу: **дата/время**, **температура**, **давление**, **PM2.5**, **PM10** и т.д.  
- Скрипт **сам ищет** колонки по заголовкам (`date`, `temperature`, `pressure`, `pm2.5`, `pm10`, …) или использует фиксированные позиции для формата **AirStationLog** (первая строка — уже данные).

### C2. Вставить скрипт

1. В таблице: **Расширения** → **Apps Script** (откроется редактор скриптов).
2. Удалите содержимое файла `Code.gs` по умолчанию.
3. Вставьте **весь** код из файла репозитория:  
   `scripts/weather-sheet-to-firebase.gs`
4. **Сохранить** (Ctrl+S), при необходимости дайте проекту имя, например «Weather to Firebase».

### C3. Свойства проекта (доступ к Firebase)

Скрипт читает URL и ключ из **свойств проекта** (не из кода — так ключ не попадёт в историю, если вы поделитесь кодом).

1. В редакторе Apps Script: **Проект** (шестерёнка) → **Свойства проекта**  
   (в англ. UI: **Project Settings**).
2. Вкладка **Свойства скрипта** / **Script properties** → **Добавить свойство скрипта** / **Add script property**.

| Свойство (ключ)   | Значение |
|-------------------|----------|
| `FIREBASE_DB_URL` | URL базы **без** слэша в конце, например `https://xxx-default-rtdb.firebaseio.com` |
| `WEATHER_DATA_KEY`| Тот же ключ, что `VITE_WEATHER_DATA_KEY` в `.env` |

Сохраните.

### C4. Первый запуск и доступ

1. В редакторе сверху выберите функцию **`syncWeatherToFirebaseFull`**.
2. **Выполнить** (Run).
3. Google запросит **разрешения**:
   - Выберите аккаунт.
   - Если «Приложение не проверено» → **Дополнительно** → **Перейти к … (небезопасно)** — это нормально для личного скрипта.
   - Разрешите доступ к внешним URL (**UrlFetchApp**) — иначе запросы в Firebase не пройдут.
4. После успешного выполнения в **Журнале выполнения** (Execution log) не должно быть ошибок Firebase.

`syncWeatherToFirebaseFull` **сбрасывает** счётчик последней строки и **отправляет все строки** листа. Дальше для регулярной работы используйте **`syncWeatherToFirebase`** — она шлёт только **новые** строки.

### C5. Триггер по времени

1. Слева в Apps Script: **Триггеры** (иконка часов).
2. **Добавить триггер**:
   - Функция: **`syncWeatherToFirebase`**
   - Источник события: **По времени**
   - Тип триггера: **По времени**
   - Интервал: например **каждые 5 минут** или **каждые 15 минут** (чем чаще — тем больше нагрузка на квоту Apps Script; раз в 1–2 минуты — разумный максимум для бесплатного аккаунта).

Сохраните. Скрипт будет дописывать в Firebase только новые строки с листа.

2. **Сайт** → Метеостанция → данные из Firebase (подпись «Данные из Firebase»).

## Если что-то не работает

| Симптом | Что проверить |
|--------|----------------|
| Скрипт: «Задайте FIREBASE_DB_URL…» | Свойства скрипта заданы и сохранены, без опечаток в именах ключей. |
| Firebase 401 / 403 | Правила Realtime Database опубликованы; URL тот же проект. |
| Скрипт не доходит до Firebase | При первом запуске выдать разрешение на доступ к внешним URL. |
| На сайте пусто | В `.env` те же `VITE_FIREBASE_DATABASE_URL` и `VITE_WEATHER_DATA_KEY`; пересборка после правок `.env`. |
| Дубли строк | Не вызывайте `syncWeatherToFirebaseFull` постоянно — только один раз или после сброса; обычный режим — `syncWeatherToFirebase` + триггер. |

## Формат записи в Firebase

Путь: `weather/{WEATHER_DATA_KEY}/rows/{autoId}`

Объект:

- `date` — timestamp в миллисекундах  
- `temperature`, `pressure`, `pm1`, `pm25`, `pm10` — числа (если есть в таблице)  
- `station` — строка (опционально)

На сайте берутся последние **10 000** точек — при записи раз в 5–15 минут этого хватает на длительный лог.

## Два листа: сырые за сутки + агрегаты по часам (max)

Когда ESP дописывает в таблицу через Sheets API, удобно разделить:

| Лист | Назначение | Как обновлять |
|------|------------|----------------|
| **Raw** (или текущий лог) | Каждая точка как приходит | Только **append** — ниже не трогаем |
| **Hourly** (третий лист) | По одной строке на час: **максимумы** за час (temp, pressure, **pm1_max, pm25_max, pm10_max** — три размера PM) | Пересчёт по расписанию из Raw |

### Зачем

- Сырой лист растёт бесконечно — в Firebase его лучше по-прежнему слать **инкрементально** (как `syncWeatherToFirebase`), а не «всю таблицу заново».
- Агрегированный лист маленький (за сутки максимум 24 строки на метрику или одна строка на час с несколькими колонками) — **его можно целиком перезаписывать** или целиком пушить в Firebase без стыда.

### Как лучше забирать «готовые» данные

1. **Не тянуть всю таблицу сырья** каждый раз — только новые строки → в Firebase как сейчас.
2. **Агрегаты** — варианты:
   - **A)** Скрипт раз в час читает с листа Raw только **диапазон за сегодня** (или за вчера), в памяти считает max по каждому часу, **перезаписывает** на листе Hourly блок за этот день (или дописывает одну строку «текущий час»).
   - **B)** Тот же скрипт после пересчёта делает **один PUT/PATCH** в Firebase, например  
     `weather/{KEY}/hourly/YYYY-MM-DD` — объект `{ "0": { temp, pm25, ... }, "1": ..., ... }`  
     или массив из 24 элементов. Тогда сайт может запрашивать **только** эту ветку для графика «по часам» — без 10k точек.
3. **«Всю таблицу заново»** — имеет смысл **только для агрегатного листа** (или для одного дня как JSON в Firebase). Для Raw — по-прежнему инкремент.

### Структура листа Hourly (пример)

Заголовки, например:

`hour_start_ms` | `date_iso` | `temp_max` | `pressure_max` | **`pm1_max`** | **`pm25_max`** | **`pm10_max`** | `count`  
или одна строка на час: первая колонка — timestamp начала часа, дальше max по полям.

Пересчёт: Apps Script по триггеру **каждый час** (или раз в 15 мин — только текущий час пересчитать):

1. Прочитать Raw за сегодня.
2. Сгруппировать по `hour = floor(date_ms / 3600000)` (в часовом поясе станции).
3. Для каждой группы — `Math.max` по нужным колонкам.
4. Записать на лист Hourly и/или в Firebase `hourly/...`.

### Итого

- **Сырьё:** append в Sheet → инкрементальный пуш в `rows` (как сейчас).
- **Агрегаты:** отдельный лист + отдельная ветка в Firebase; **перезапись целого дня/всей таблицы агрегатов** — нормальная практика, объём маленький.
- Сайт потом может сначала показывать «по часам» из `hourly`, а детальный график — из `rows` за последние N точек.

## Три листа в одной книге (сырьё → 24 ч → почасовые + очистка)

Если ESP пишет в **один** лист и он раздувается:

1. **`scripts/weather-sheets-aggregate.gs`** — второй скрипт в тот же проект Apps Script (или отдельный файл в том же проекте).
2. Листы (имена по умолчанию, можно переименовать в свойствах скрипта):
   - **Raw** — куда дописывает ESP (как сейчас).
   - **Last24h** — перезаписывается: только строки за последние 24 часа (полные строки как в Raw).
   - **Hourly** — почасовые **максимумы** + счётчик точек за час; компактное долгосрочное хранение.
3. **Очистка Raw** — после того как Hourly обновлён, скрипт удаляет с листа Raw строки старше **RETENTION_DAYS** (по умолчанию 7 дней), чтобы вес книги не рос бесконечно.

Триггер: функция **`runWeatherSheetsPipeline`** по времени (например раз в час). Порядок внутри: Hourly → Last24h → cleanup Raw.

### Дата без времени в логе (`10.03.2026` без часов)

Если ESP пишет только дату, при разборе получалось 00:00 — строка **выпадала** из окна «последние 24 часа». Сейчас в **`rebuildLast24hFromRaw`**:

- есть **время** в ячейке → отбор как раньше: `timestamp` в интервале \[сейчас−24ч, сейчас\];
- **только дата** → в Last24h попадают строки за **сегодня и вчера** по календарю (локальное время скрипта).

Чтобы именно «скользящие 24 часа», в лог лучше дописывать время (`DD.MM.YYYY HH:mm`).

### Автодобавление в Firebase после пайплайна

Новые строки с **листа Raw** по-прежнему дописываются в Firebase **инкрементально** (`WEATHER_LAST_ROW`). После обновления Last24h/Hourly пайплайн вызывает **`syncWeatherToFirebaseFromRawSheet`** (если в проекте есть `weather-sheet-to-firebase.gs` и заданы `FIREBASE_DB_URL` / `WEATHER_DATA_KEY`). Тогда **вручную активный лист менять не нужно** — пуш идёт с Raw по имени листа из `RAW_SHEET_NAME`.

### Заполнить Sheet2 и Sheet3 прямо сейчас (без ожидания триггера)

Отдельного HTTP API в репо нет — всё в Apps Script. Чтобы листы **начали заполняться сразу**:

1. **Меню в таблице** (рекомендуется)  
   После сохранения скрипта **обнови вкладку с таблицей (F5)**. В шапке появится меню **«Метео»**:
   - **Заполнить Sheet2 и Sheet3 сейчас** — только Hourly + Last24h, **без** удаления строк с Sheet1.
   - **Полный цикл** — то же + очистка Sheet1 по `RETENTION_DAYS`.

2. **Вручную в редакторе скриптов**  
   Выбрать функцию **`runWeatherSheetsFillOnly`** → Выполнить.

3. **Web App как «API»**  
   - Свойство скрипта `WEBAPP_SECRET` = случайная строка.  
   - Deploy → New deployment → Type **Web app** → Execute as **Me** → Who has access **Anyone** (или ограниченно).  
   - Открыть в браузере:  
     `https://script.google.com/macros/s/.../exec?key=ТВОЙ_WEBAPP_SECRET`  
   В ответ `ok: ...` — листы 2 и 3 обновлены.

Свойства скрипта (опционально): `RAW_SHEET_NAME`, `SHEET_24H_NAME`, `SHEET_HOURLY_NAME`, `RETENTION_DAYS`.

Дальше Firebase можно кормить **с Last24h** (мало строк) или **с Hourly** — по желанию отдельным маленьким скриптом POST в тот же Firebase.

## Формат листа без шапки (первая строка — данные)

Пример:

```text
26.01.2026 14:55:27    26.01.2026 14:55:27    Searching...    (пусто)    16    22    22    nan    nan
```

| Колонка | Смысл |
|--------|--------|
| **A, B** | Дата/время `DD.MM.YYYY HH:mm:ss` (дубликат) — в скрипте берётся колонка A |
| **C** | Статус (Searching…) — в Firebase как station **не** пишется |
| **D** | Пусто |
| **PM в логе ESP** | В исходнике с ESP порядок колонок (0-based): **PM1 — индекс 6, PM2.5 — 7, PM10 — 9** → в скриптах те же поля в Firebase и на листе Hourly (`pm1` / `pm25` / `pm10` и `*_max`) |
| **Далее** | `nan` — пропускаются; когда появятся GPS, **температура, влажность** — либо добавьте строку заголовков над данными, либо правьте индексы в `getColumnIndices` (`isStationLogRowFormat`) в обоих `.gs` |

Парсинг даты в Apps Script сделан вручную под `DD.MM.YYYY`, чтобы не получить Invalid Date.
